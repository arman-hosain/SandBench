# SandBench

Benchmark framework for evaluating LLMs on malware analysis using VirusTotal sandbox reports.
Each mode asks the LLM to extract MITRE ATT&CK TTPs, IOCs, and the malware family from a VT report.
Results are scored against CAPA-extracted ground truth.

---

## Modes

| Mode | What it does |
|------|-------------|
| A | Single-shot — one prompt, one response. Baseline. |
| B | Judge-refined — Mode A output fed into a judge loop (up to N iterations) |
| C | Agentic — ReAct loop; the model calls tools to query the report step by step |
| D | Agentic + Judge — Mode C followed by Mode B's judge refinement |

---

## Project Structure

```
sandbench/
├── config.json                 # All runtime settings — edit this, not the code
├── preprocess.py               # Convert raw VT JSONs → clean LLM inputs + ground truth
├── run_benchmark.py            # Main runner — orchestrates all experiments
├── modes/
│   ├── mode_a.py               # Single-shot
│   ├── mode_b.py               # Judge-refined
│   ├── mode_c.py               # Agentic (ReAct)
│   └── mode_d.py               # Agentic + Judge
├── tools/
│   ├── vt_environment.py       # Tool interface for the agent (Modes C/D)
│   └── llm_utils.py            # Safe LLM call wrapper
├── eval/
│   └── vt_metrics.py           # TTP F1, IOC recall, family accuracy, composite score
├── generate_graphs.py          # Base comparison charts
├── generate_extended_charts.py # Extended charts (boxplots, efficiency frontier, table)
├── generate_log_viewer.py      # Interactive HTML log viewer

├── dataset/
│   ├── vt_high_speed_results/  # Raw VT JSON files (one per sample)
```

## Step 1 — Preprocess

Run once before benchmarking. Reads raw VT JSONs, extracts ground truth, and writes clean
text reports that get fed to the LLM.

```bash
python preprocess.py                   # process all samples
python preprocess.py --max 100         # first 100 only (good for a quick check)
python preprocess.py --stats           # print dataset stats, write nothing
python preprocess.py --hash <sha256>   # single sample
```

Output: `dataset/preprocessed/<hash>.json` and `dataset/ground_truth.json`.

---

## Step 2 — Run the benchmark

```bash
# All models, all modes, all preprocessed samples
python run_benchmark.py

# Quick test — 10 samples, Mode A only, one model
python run_benchmark.py --max-samples 10 --modes A --models gpt-4o

# Specific modes and models
python run_benchmark.py --modes A B --models llama-3.1-8b qwen3-8b

# Single sample by hash
python run_benchmark.py --hash 000050fb...

# Print what would run, no LLM calls made
python run_benchmark.py --dry-run
```

Results are written to `results/benchmark_results.json`.
Per-experiment logs go to `logs/<hash>_<mode>_<model>.log.json`.

---

## Step 3 — Generate charts

```bash
python generate_graphs.py
python generate_extended_charts.py
```

## Step 4 — Generate tables

Produces Markdown and LaTeX versions of all six result tables:

| Table | File stem | Contents |
|-------|-----------|----------|
| 01 | `01_composite_score` | Composite Score by Model × Mode (mean ± std) |
| 02 | `02_metric_breakdown` | Per-model breakdown: TTP F1, TTP Recall, Verdict Acc., Composite across all modes |
| 03 | `03_all_conditions` | All metrics × all Model × Mode conditions (heatmap data) |
| 04 | `04_cost_latency` | Average LLM API calls and inference latency per mode |
| 05 | `05_improvement_delta` | Mode A vs Mode D scores and Δ per metric per model |
| 06 | `06_verdict_accuracy` | Threat Verdict Accuracy by Model × Mode (mean ± std) |

```bash
python generate_tables.py
```

Output goes to `tables/` — each table gets a `.md` and a `.tex` file:

```
tables/
├── 01_composite_score.md / .tex
├── 02_metric_breakdown.md / .tex
├── 03_all_conditions.md / .tex
├── 04_cost_latency.md / .tex
├── 05_improvement_delta.md / .tex
└── 06_verdict_accuracy.md / .tex
```

Custom paths:

```bash
python generate_tables.py --results ./results/benchmark_results.json --output ./tables
```

The LaTeX tables use `booktabs` (`\toprule`, `\midrule`, `\bottomrule`) — add `\usepackage{booktabs}` to your document preamble.

## config.json Reference

### Paths

```json
"dataset_dir":       "./dataset/vt_high_speed_results",
"preprocessed_dir":  "./dataset/preprocessed",
"ground_truth_file": "./dataset/ground_truth.json",
"results_dir":       "./results",
"log_dir":           "./logs"
```

Change these if your data lives elsewhere. All other paths in the code derive from these.

---

### API Keys and Endpoints

```json
"openai_api_key":      "sk-proj-...",
"local_vllm_base_url": "http://localhost:8002/v1",
"local_vllm_api_key":  "EMPTY"
```

- `openai_api_key` — needed only for models with `"client": "openai"`.
- `local_vllm_base_url` — URL of your local vLLM server (OpenAI-compatible).
- `local_vllm_api_key` — leave as `"EMPTY"`; vLLM ignores the key.

---

### Models

```json
"models": {
  "my-model-key": {
    "api_name":       "org/model-name",
    "client":         "local",
    "context_window": 32768
  }
}
```

| Field | Required | Description |
|-------|----------|-------------|
| `api_name` | Yes | Model ID passed to the API — HuggingFace path for local, OpenAI name for cloud |
| `client` | Yes | `"local"` uses `local_vllm_base_url`; `"openai"` uses the OpenAI API |
| `context_window` | No | Per-model override of `llm.context_window`. Set this when the model supports more than the default 8192 tokens (e.g. `128000` for GPT-4o) |

**To add a new model**, add an entry under `"models"` and reference it with `--models`:

```json
"mistral-7b": {
  "api_name": "mistralai/Mistral-7B-Instruct-v0.3",
  "client": "local"
}
```

```bash
# Start vLLM with that model, then:
python run_benchmark.py --models mistral-7b --modes A B
```

### LLM Settings

```json
"llm": {
  "max_tokens":             2048,
  "temperature":            0.3,
  "context_window":         8192,
  "seed_report_chars":      3000,
  "observation_chars":      3000,
  "synthesis_report_chars": 1500,
  "refine_report_chars":    3000,
  "prev_output_chars":      2000
}
```

| Field | Used by | Description |
|-------|---------|-------------|
| `max_tokens` | All | Max tokens the LLM can generate per call |
| `temperature` | All | Sampling temperature. `0` = deterministic, higher = more varied |
| `context_window` | Mode A | Total token budget for the model. Controls how much of the report gets sent. Per-model `context_window` takes priority over this value. |
| `seed_report_chars` | C, D | Characters of the report shown to the agent at the start of the investigation loop |
| `observation_chars` | C, D | Max characters of each tool call result shown to the agent |
| `synthesis_report_chars` | C, D | Characters of the report included in the final synthesis prompt |
| `refine_report_chars` | B, D | Characters of the report included in the judge refinement prompt |
| `prev_output_chars` | B, D | Characters of the previous LLM analysis shown to the judge and refiner |

**Scaling up for a larger context window:**
If your model supports 32k or 128k tokens, increase the `*_chars` values proportionally.
A rough guide: if `context_window` goes from 8192 → 32768 (4×), you can raise `seed_report_chars`
from 3000 → 12000, `observation_chars` from 3000 → 12000, etc.

```json
"llm": {
  "context_window":         32768,
  "seed_report_chars":      12000,
  "observation_chars":      12000,
  "synthesis_report_chars": 6000,
  "refine_report_chars":    12000,
  "prev_output_chars":      8000
}
```

---

### Agent Settings

```json
"agent": {
  "budget": 15
}
```

`budget` — max number of tool calls per sample in Modes C and D.
`get_sample_overview` is free (cost 0); all other tools cost 1.
Higher budget = more thorough investigation, more LLM calls, higher latency.

---

### Judge Settings

```json
"judge": {
  "max_iterations": 5
}
```

`max_iterations` — how many times the judge can send feedback and the analyst can revise (Modes B, D).
The loop exits early if the judge finds no gaps. Set to `1` for a single review pass.

---

### Evaluation Settings

```json
"eval": {
  "ttp_top_level_only": true,
  "composite_weights": {
    "non_agentic": {
      "ttp_f1":          0.50,
      "ioc_recall":      0.35,
      "family_accuracy": 0.15
    },
    "agentic": {
      "ttp_f1":               0.40,
      "ioc_recall":           0.30,
      "family_accuracy":      0.10,
      "evidence_grounding":   0.20
    }
  }
}
```

| Field | Description |
|-------|-------------|
| `ttp_top_level_only` | `true` collapses sub-techniques (T1059.001 → T1059) before scoring. Keep `true` unless your ground truth has sub-technique labels. |
| `composite_weights` | How much each metric contributes to the final score. Weights are automatically re-normalised when a sample has no ground truth for a dimension (e.g. no known family). |

---

### Preprocess Settings

```json
"preprocess": {
  "max_registry_keys": 20,
  "max_dropped_files": 30
}
```

Controls how many items from each category are included in the preprocessed report text.
Raise these if you have a larger context window and want the model to see more artifacts.

---

## Metrics

| Metric | Modes | Description |
|--------|-------|-------------|
| TTP F1 | All | Set-level F1 between predicted and CAPA-extracted MITRE technique IDs |
| IOC Recall | All | Per-type recall over IPs, domains, files, registry keys, mutexes |
| Family Accuracy | All | Exact match (with alias handling) on malware family name |
| Evidence Grounding | C, D | Fraction of agent-cited IOCs that actually exist in ground truth |
| Composite Score | All | Weighted combination of the above |

## Common Recipes

```bash
# Cheapest/fastest baseline — Mode A only, all models
python run_benchmark.py --modes A

# Full comparison on 50 samples
python run_benchmark.py --max-samples 50

# Debug a single sample across all modes
python run_benchmark.py --hash <sha256> --modes A B C D

# Test config without spending tokens
python run_benchmark.py --dry-run
```
