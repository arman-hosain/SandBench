# CyFER-Bench — Vulnerability Explanation Module

**CyFER-Bench Tasks covered:** (4) Weakness Identification · (5) Vulnerable Code Localization and Exploit Inference

Evaluates how well LLMs can explain C/C++ vulnerabilities flagged by static analysis (Flawfinder), using the NIST Juliet Test Suite. The model receives only a code snippet with the flagged line — no CWE ID and no Flawfinder rationale — and must identify and explain the vulnerability.

---

## Project Structure

```
vuln_explanation/
├── build_dataset.py           # Parse Flawfinder reports + .cpp files → dataset.json
├── generate_explanations.py   # Query LLMs for vulnerability explanations
├── evaluate.py                # Score explanations on 5 axes
└── dataset/
    ├── dataset.json           # Built dataset (generated by build_dataset.py)
    ├── CWE78/                 # OS Command Injection — code/ and report/ subdirs
    ├── CWE134/                # Uncontrolled Format String
    └── CWE190/                # Integer Overflow
```

---

## Dataset Structure Expected

```
dataset/
  CWE78/
    code/    ← .cpp files (Juliet test cases)
    report/  ← Flawfinder report files (one per .cpp)
  CWE134/  ...
  CWE190/  ...
```

## Pipeline

### Step 1 — Build the Dataset JSON

```bash
python build_dataset.py \
  --dataset dataset \
  --output dataset/dataset.json \
  --context 10        # lines before/after flagged line
```

### Step 2 — Generate LLM Explanations

```bash
# Zero-shot baseline
python generate_explanations.py \
  --dataset dataset/dataset.json \
  --model gpt-4o \
  --api-key $OPENAI_API_KEY \
  --prompt-mode zero_shot \
  --output-dir explanations

# Few-shot
python generate_explanations.py --prompt-mode few_shot ...

# Ablation: provide CWE ID as hint
python generate_explanations.py --prompt-mode hint ...

# vLLM-hosted local model
python generate_explanations.py \
  --model meta-llama/Llama-3.1-70B-Instruct \
  --api-base http://localhost:8001/v1 \
  --api-key EMPTY \
  --prompt-mode zero_shot ...
```

### Step 3 — Evaluate

```bash
python evaluate.py \
  --results explanations/gpt-4o/zero_shot/results.json \
  --output  explanations/gpt-4o/zero_shot/scores.json \
  --judge-model gpt-4o \
  --api-key $OPENAI_API_KEY
```

---

## Evaluation Axes

| Axis | Method | Weight |
|------|--------|--------|
| CWE Alignment | Rules-based keyword match | 25% |
| Construct Match | Rules-based exact/partial | 20% |
| Completeness | LLM-as-judge (root cause + impact + fix) | 25% |
| Severity Calibration | LLM-as-judge (urgency language vs. score 1–4) | 15% |
| No Hallucination | LLM-as-judge (inverted hallucination score) | 15% |

Final score = weighted average over all 5 axes (0.0–1.0).

---

## What the LLM Sees vs. Ground Truth

**LLM input:** code snippet (±N lines around flagged line, with `>>>>` marker), flagged line number — no CWE ID, no Flawfinder rationale (in zero-shot and few-shot modes).

**Ground truth (never shown to LLM):** Flawfinder rationale, CWE ID, severity, dangerous construct name.

---

## Notes

- Label leakage is automatically masked: Juliet filenames like `CWE78_OS_Command_Injection__char_execl_33.cpp` are redacted before being shown to the model.
- Use `--context 15` or `--use-full-function` for CWE-78 where source and sink are far apart in the code.
